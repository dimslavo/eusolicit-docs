## Part I â€” Epic 13 Story Review (7-17 Completed; 7 Not Started)

### Story 7-17: Proposal Generate Security Audit Hardening

**Context:** Carried from E07 as a P0/BLOCKER NFR FAIL (NFR4 security + NFR12 compliance). Triggered by SSE error events leaking AI-Gateway stack traces to authenticated tenants and missing `shared.audit_log` entries for `POST /generate`.

**What went well:**

1. **[PATTERN] NFR hot-fix + proper story workflow** â€” A minimal inline commit (`2d41fcf`) unblocked the NFR halt on the same day the assessment ran. The full-spec implementation story was created in parallel. This two-step pattern (emergency inline patch â†’ proper story for hardening) is the correct NFR FAIL response and avoids blocking subsequent epics while the proper implementation is tracked. Reference: E07 retro Security Hardening pattern.

2. **[PATTERN] Adversarial review catches behavioural regressions** â€” The 3rd review pass (2026-04-23) correctly identified that 3 integration tests were failing at HEAD after previously being presented as GREEN. Without re-execution of the test suite, the regression would have shipped. The adversarial review system is functioning as intended.

3. **[PATTERN] asyncio.CancelledError requires explicit handling in Python 3.8+** â€” `CancelledError` inherits from `BaseException`, not `Exception`, in Python 3.8+. A bare `except Exception` block silently drops cancellation signals. The fix (explicit `except asyncio.CancelledError`) is now documented and should be added to the backend story template as a required check for any coroutine with `async with / finally` blocks.

4. **[PATTERN] structlog requires stdlib routing for pytest caplog** â€” `structlog.PrintLoggerFactory()` bypasses the Python stdlib logging machinery. `pytest`'s `caplog` fixture captures stdlib `logging.Logger` records only. Fix: session-scoped `configure_structlog_stdlib` autouse fixture in `conftest.py` routes structlog through `stdlib.LoggerFactory()`. This is now a required fixture in every service's `tests/conftest.py`.

5. **[PATTERN] Migration spec contradiction is a predictable failure mode** â€” Story header claimed "No schema migration needed" but migration 025 was required because `shared.audit_log` was missing `company_id` and `correlation_id` columns. This type of spec contradiction (schema assumptions made before checking ORM models) has occurred in multiple epics. Spec writers must grep the target schema before writing "no migration needed."

**What needs improvement:**

1. **[ANTI-PATTERN] Dev session regression between review passes**
   - `[ACTION]` IMPACT: `standards_update` SEVERITY: `critical`
   - Between the 2nd-pass approval (2026-04-18) and the 3rd review (2026-04-23), the endpoint was amended to write **two** audit rows per `POST /generate` â€” one synchronous "entry" row at `proposals.py:723-734` plus the fire-and-forget terminal row from the background task. This directly contradicts AC3 ("exactly one row") and reverted the F4 resolution from the prior pass.
   - Root cause: a new dev session modified the endpoint for unrelated reasons (AC18 from a sibling story), overwriting approved code without awareness that the prior review had explicitly resolved the duplicate-write pattern.
   - Fix: Before any dev session modifies a file touched in a previous review, the agent MUST re-read the current file state and cross-check it against the prior review's resolved findings. Add to dev-story template: "Before editing a file with prior review findings, re-read and list all resolved deviations that must be preserved."

2. **[ANTI-PATTERN] Review approval without test re-execution**
   - `[ACTION]` IMPACT: `standards_update` SEVERITY: `critical`

   - Fix: Any review that includes the phrase "all tests passing" or "PASS" for a previously-failing test MUST include the actual test execution summary line (e.g., `3 failed, 16 passed`) in the review artifact. Without a quoted execution line, the approval is invalid. Add to code-review template: "Test verification: quote actual pytest output line. No approvals without quoted execution result."

3. **[ANTI-PATTERN] AC text not updated to reflect implementation-time schema choices**

Future reviewers will rediscover this as a spec violation. Fix: Any implementation-time schema deviation from an AC must update the AC text in the story file before the review approves. AC text is the contract, not documentation.

4. **[ANTI-PATTERN] Sprint-status shows `done` for a story with blocking review findings**

## Part II â€” Learnings for Subsequent Epics

### Process Learnings

**[PATTERN] Coordinator story pattern for cross-cutting hardening**
Epic 13 introduced a coordinator story (`drift-recovery-story`) that owns end-to-end acceptance across multiple sub-stories. The coordinator holds AC4 (Stripe circuit-breaker) and AC5 (billing metrics) as direct work, while delegating AC1â€“AC3 to `inj-01/02/03`. This is the right architecture for multi-concern hardening epics: sub-stories deliver narrow slices, the coordinator owns the gate. Apply this pattern to any future hardening epic with >3 independent work streams.

**[PATTERN] Hot-fix commit reference in story Dev Notes**
Story 7-17's "Hot-Fix Context" section (commit `2d41fcf`) specifying exactly what was pre-applied vs. what remains â€” with file paths and line numbers â€” is the reference format for all stories that build on a prior emergency fix. This section prevents dev agents from re-implementing already-merged changes.

**[ANTI-PATTERN] Carry-forward items require execution-order enforcement**
- `[ACTION]` IMPACT: `config_tuning` SEVERITY: `critical`
- Injected stories for Dependabot (inj-01), k6 (inj-02), and TEA reviews (inj-03) have been in `ready-for-dev` since 2026-04-25 with no dev session started. This is the same failure mode documented in E08 retro: "Retrospective [ACTION] items not executed in subsequent epic â€” retro-to-action feedback loop is broken." Injected stories must be the FIRST items processed in the next epic, not appended to the queue. The orchestrator config must set `inj-*` story priority to `p0` (before any new feature work).

**[ANTI-PATTERN] Stories injected into wrong epic scope**
- `[ACTION]` IMPACT: `config_tuning` SEVERITY: `medium`
- `dw-01`, `dw-02`, `dw-03` all state `Epic: Post-E07 cleanup (no assigned epic)` in their headers, yet they are tracked in Epic 13. The story file origin header and the sprint-status epic assignment are misaligned. Fix: when a story is assigned to an epic, its header must be updated with the assigned epic number. Misaligned headers cause confusion about when the story's AC scope was defined.

### Technical Learnings

**[PATTERN] Fire-and-forget audit write pattern (E13 canonical form)**
The Story 7-17 implementation established the authoritative form for fire-and-forget audit writes:
```python
asyncio.create_task(_write_generation_audit(
    session=session,
    proposal_id=...,
    user_id=...,
    company_id=...,
    ip_address=...,
    correlation_id=...,
    before_snapshot=...,
    # after_outcome set in _run_generation_task.finally
))
```
The task creates its own `async with session.begin()` and has an internal `try/except Exception` that logs `audit_write_failed` at ERROR without propagating. This is now the canonical form for all audit-write fire-and-forget calls across all services. Prior forms using `await write_audit_entry()` + `await session.commit()` inline are PROHIBITED on the TTFB path (Rule 45).

**[PATTERN] Single-point sanitization helper for all error paths**
`_sanitize_generation_error(exc, *, correlation_id, stream_incomplete=False)` centralizes all SSE error frame construction. No SSE error frame may be constructed outside this function (AC2). Applying this pattern to other SSE proxy endpoints: any endpoint that emits error frames must have a single helper that (a) produces a stable machine-code schema, (b) never includes raw exception fields, (c) injects `correlation_id`. The `stream_incomplete` kwarg pattern handles the boundary case without a new code path.

**[ANTI-PATTERN] Schema `Literal` type for status fields**
- `[ACTION]` IMPACT: `prompt_adjustment` SEVERITY: `medium`
- `ProposalDetailResponse.status` was bare `str` instead of `Literal["draft", "active", "archived"]`. This prevents OpenAPI enum schema generation and allows unknown values to reach the frontend without validation errors. All Pydantic response schemas with enumerable string fields MUST use `Literal[...]` or `StrEnum`. Add to backend story template: "All response schema string fields with known value sets MUST use `Literal` or `StrEnum` â€” never bare `str`."

**[ANTI-PATTERN] `asyncio.get_event_loop()` deprecated in Python 3.12**
- `[ACTION]` IMPACT: `prompt_adjustment` SEVERITY: `medium`
- DW-02 documents that `asyncio.get_event_loop()` is deprecated in Python 3.12 and must be replaced with `asyncio.new_event_loop()`. Any Celery task that bridges syncâ†’async execution must use `asyncio.new_event_loop().run_until_complete()` in a `try/finally loop.close()` block â€” never `asyncio.run()` (fails with running loop) and never `asyncio.get_event_loop()` (deprecated). Add to Celery task template.
