## 5. Key Insights and Learnings

### 5.1 Null vs. Empty List Is a First-Class API Design Choice

The gantt_data null/empty distinction in S11.07 codifies a pattern that applies across all optional complex response fields: `null` signals absence (don't render the component), `[]` signals presence with no content (render empty state). This distinction must be preserved through all parsing and serialization layers.

**Standard:** For any response field that is a list AND whose absence has different UI semantics than an empty list, the field type must be `list | None` (not `list`), with explicit `null` on absence and `[]` on empty. This distinction must appear in the AC, the Pydantic schema, the parser function, and the frontend component's conditional render. `[PATTERN]`

### 5.2 Migration Downgrade Safety Requires Multi-Row Data Guards

S11.01's downgrade bug only surfaces in production data (multiple ESPD profiles per company). The fix — `ROW_NUMBER() OVER (PARTITION BY company_id ORDER BY created_at)` — assigns unique version numbers before re-adding the unique constraint. Unit tests with single rows per company miss this entirely.

**Standard:** Any migration downgrade that recreates a UNIQUE or PK constraint must include a regression test that inserts ≥2 rows with the same grouping key before downgrading. The test asserts the downgrade completes without constraint violations. `[PATTERN]`

### 5.3 The "Done = Stub" Anti-Pattern Breaks TRACE_GATE

S11.07 was marked done with frontend stub API functions and zero GREEN backend tests. TRACE_GATE correctly identified the gap. The root cause: sprint-status.yaml status was set before implementation verification.

**Standard:** A story cannot be marked `done` in sprint-status.yaml if: (a) any frontend component in the story references a stub API function for that story's endpoints, OR (b) the TRACE_GATE coverage for that story's ACs shows 0 tests. Both conditions must be verified at done-marking time. `[ANTI-PATTERN]`

### 5.4 i18n Compliance Requires Automated Enforcement, Not Review Discipline

The repeated i18n findings across S11.11, S11.13, S11.14 prove that human review cannot consistently catch hardcoded strings. Three stories, three epics' worth of i18n discipline, and still strings are missed when developers add them during fix passes.

**Standard:** An ESLint rule enforcing `no-literal-text` in JSX (or equivalent `i18n-linter`) must be configured for `apps/client` and `apps/admin`. Hardcoded display strings are a lint error, not a review finding. `[ACTION]` `IMPACT: standards_update` `SEVERITY: high`

### 5.5 Design System Component Violations Need ATDD-Level Detection

S11.12's native `<select>` instead of `<Select>` from `@eusolicit/ui` is a design consistency failure that requires the developer to know which AC-specified components are design system components vs. HTML natives. This is domain knowledge that the reviewer must supply in review — too late.

**Standard:** Any AC specifying a component from `@eusolicit/ui` must have a corresponding ATDD source-inspection assertion: `assert "from @eusolicit/ui import" in component_source` or `assert "<Select" in component_source and "import { Select }" in component_source`. Add this to the ATDD checklist template for all frontend stories with `<Select>`, `<Dialog>`, `<Sheet>`, `<Tabs>` components. `[PATTERN]`

### 5.6 Agent Integration Pattern Is Complete and Stable

Six client-api agent endpoints and two admin-api agent endpoints all follow the same pattern: `AiGatewayClient.run_agent(agent_name, payload)`, flat 503 body on timeout/5xx, X-Caller-Service header, configurable timeout, no client-side retry. This is the complete, tested, stable pattern for E12+ agent integrations.

**Standard:** All future agent integrations MUST use this pattern. A deviation (different error body shape, client-side retry, missing X-Caller-Service header, direct AIGW URL construction) is a blocker in review. The pattern is documented in S11.04 and S11.09 as canonical references. `[PATTERN]`
