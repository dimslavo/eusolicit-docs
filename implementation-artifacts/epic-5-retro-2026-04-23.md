## What Went Well

### [PATTERN] Two-Layer Resilience Pattern Correctly Inherited from E04
`IMPACT: standards_update | SEVERITY: low`

The E04 circuit-breaker-wraps-retry pattern was faithfully applied in S05.03: `circuit.call(lambda: with_retry(_make_request, agent_name=...))`. Circuit opens after 5 consecutive failures, half-opens after 60 s. Retry applies exponential backoff (1s→2s→4s ±25% jitter), max 3 attempts. `CircuitOpenError` and 4xx responses are non-retryable at the retry layer. Applied across all 3 crawlers + enrichment worker.

### [PATTERN] 3-Queue Celery Isolation Prevents Starvation
`IMPACT: standards_update | SEVERITY: medium`

Dedicated queues: `pipeline_crawl` (crawl tasks, cleanup, publish), `pipeline_scoring` (scoring sub-tasks), `pipeline_guides` (guide generation sub-tasks). `worker_prefetch_multiplier=1` + `task_acks_late=True` for fair dispatch and no silent loss. Scoring and guide-generation load cannot starve crawl tasks. This pattern should be applied to any epic that uses Celery chains/chords with heterogeneous work types.

### [PATTERN] Atomic `INSERT ... ON CONFLICT` Upsert Eliminates Dedup Race
`IMPACT: standards_update | SEVERITY: medium`

Single-statement `INSERT INTO pipeline.opportunities ... ON CONFLICT (source_id, source_type) DO UPDATE SET ...` in `_upsert.py`. DB-level UNIQUE constraint as backstop. Dedup test (E05-P2-014) verifies zero new inserts on second identical run. No SELECT-then-INSERT race condition possible.

### [PATTERN] `SELECT FOR UPDATE SKIP LOCKED` for Concurrent Queue Workers
`IMPACT: standards_update | SEVERITY: medium`

`process_enrichment_queue` uses `SELECT ... FOR UPDATE SKIP LOCKED` to prevent two worker replicas from processing the same enrichment item concurrently. Items marked `processing` before AI Gateway calls; restored to `pending` on failure (<3 attempts) or `failed` (≥3 attempts). Correct PostgreSQL primitive for this pattern.

### [PATTERN] Dedicated `CollectorRegistry` for Prometheus Metrics
`IMPACT: standards_update | SEVERITY: low`

`PIPELINE_METRICS_REGISTRY = CollectorRegistry()` in `metrics.py` — all four metrics registered to this dedicated registry, not the default global one. Prevents `ValueError: Duplicated timeseries in CollectorRegistry` when pytest re-imports modules across test session. `/metrics` endpoint calls `generate_latest(PIPELINE_METRICS_REGISTRY)`. Apply this pattern to any service adding Prometheus instrumentation.

### [PATTERN] Celery Eager Mode for Full-Chain Integration Tests
`IMPACT: standards_update | SEVERITY: low`

`task_always_eager=True` + testcontainers PostgreSQL + respx mock router allows the complete Beat→crawl→score→guide→publish chain to run in a single process without a real broker. E05-P0-009 verifies all five chain steps (crawl counts, opportunity rows, relevance_scores JSONB, submission_guides rows, Redis Stream event payload). CI-safe, no external credentials.

### [PATTERN] Pagination Sentinel Normalization (TED)
`IMPACT: standards_update | SEVERITY: low`

TED pagination termination uses a falsy check (`if raw_token:`) rather than explicit None comparison. Covers `None`, `""`, `0`, and missing-key (`.get()` default). Prevents infinite loops when agents return empty-string tokens instead of null. Documented as E05-R-004 mitigation. Applicable to any pagination pattern where the "end of pages" sentinel may vary by agent version.

### TRACE_GATE PASS on First Run — 100% Coverage at All Priorities
`IMPACT: standards_update | SEVERITY: low`

42 acceptance criteria, 100% FULL coverage. 21 P0, 16 P1, 5 P2 criteria — all mapped to at least one test. 102 tests total. No critical gaps. Consistent with E04's TRACE_GATE PASS — the testcontainers + respx stack enables thorough coverage without external dependencies.

## What Could Be Improved

### [ANTI-PATTERN] `_RUN_ID_REGISTRY` In-Memory Dict — Incompatible with Celery Prefork Pool
`IMPACT: story_injection | SEVERITY: high`

```
[ACTION] Replace _RUN_ID_REGISTRY module-level dict with Redis SETEX+GET in crawl_aop.py, crawl_ted.py, crawl_eu_grants.py
IMPACT: story_injection
SEVERITY: high
```

The `_RUN_ID_REGISTRY: dict[str, str]` is a module-level Python dict in all three crawler tasks. In Celery prefork production mode, the `task_failure` signal fires in a different OS process than the task body — `_pop_run(task_id)` returns `None`, the `crawler_runs` record is never marked `failed`, and orphaned `running`/`retrying` rows accumulate silently. Tests pass because they run in eager mode (single process), masking the race.

**Fix:** Replace `_register_run(task_id, run_id)` with `redis.setex(f"pipeline:run_id:{task_id}", ttl=3600, run_id)` and `_pop_run(task_id)` with `redis.getdel(f"pipeline:run_id:{task_id}")`. Applies to all three crawler tasks.

This gap is confirmed by the NFR assessment as HIGH priority item #1.

### [ANTI-PATTERN] Non-Atomic Transaction Pattern — Partial-Failure Window in Crawler Tasks
`IMPACT: story_injection | SEVERITY: high`

```
[ACTION] Consolidate CrawlerRun creation, upsert, and status-update into minimal session boundaries; add startup recovery sweep for stuck crawler_runs
IMPACT: story_injection
SEVERITY: high
```

Crawler tasks use 3 separate `get_sync_session()` contexts: (1) `CrawlerRun` creation, (2) upsert commit, (3) `CrawlerRun` status update. A crash between (2) and (3) leaves opportunities durably persisted but `crawler_runs.status = 'running'` forever. Similarly in `process_enrichment_queue`, the enrichment result write and `session.delete(item)` are separate commits — a crash leaves items stuck in `processing` permanently with no TTL recovery. No automatic sweeper exists.

**Fix:** Add a startup recovery sweep (Celery Beat task, every 15 min) that resets `processing` items older than 10 minutes back to `pending`, and marks `running`/`retrying` `CrawlerRun` records older than the max retry window as `failed`.

### [ANTI-PATTERN] Circuit Breaker Broad `except Exception` Counts 4xx as Failures (OBS-001)
`IMPACT: prompt_adjustment | SEVERITY: medium`

`circuit_breaker.py:84` catches all exceptions including `httpx.HTTPStatusError` for 4xx responses, incrementing the failure counter. Repeated misconfigured requests (e.g., bad normalization payload producing 400) can prematurely open the circuit, blocking legitimate crawl requests. Additionally, no `# noqa: BLE001` annotation — will fail ruff BLE001 linting.

**Fix:** Add status code check in `CircuitBreaker.call()`: only increment failure count for 5xx and transport errors; raise 4xx immediately without counting as a circuit failure.

### [ANTI-PATTERN] `ENRICHMENT_BATCH_SIZE=0` Guard Missing
`IMPACT: prompt_adjustment | SEVERITY: low`

`process_enrichment_queue.py:91` computes `batch_size = int(os.environ.get("ENRICHMENT_BATCH_SIZE", "20"))` with no guard against 0 or negative values. `ENRICHMENT_BATCH_SIZE=0` silently processes nothing, leaving the enrichment queue to grow unboundedly.

**Fix:** `batch_size = max(1, int(os.environ.get("ENRICHMENT_BATCH_SIZE", "20")))`.

### [ANTI-PATTERN] Post-Dev Client.py Correction Indicates Missed Dev Discipline
`IMPACT: prompt_adjustment | SEVERITY: medium`

Story 5.5 required a post-dev correction: the dev agent broadened `except CircuitOpenError as exc:` to `except Exception as e:` in `ai_gateway_client/client.py`, masking `AIGatewayUnavailableError` from `with_retry`. Two unit tests failed on review (`test_unavailable_error_after_retries_exhausted`, `test_retry_on_timeout_exhausted`). A narrow except clause was restored and all 55 tests passed.

This reveals a pattern: scope-cleanup passes that reorganise exception handling require targeted regression tests asserting the specific exception type that surfaces to callers. The E04 anti-pattern "scope-cleanup PRs must include header contract regression tests" extends to exception surface contracts.

## Process Learnings

### [PROCESS_CHANGE] `SELECT FOR UPDATE SKIP LOCKED` Is the Standard Pattern for Celery Batch Queue Workers
Any Celery Beat task that polls a DB table for pending work and processes items in batches must use `SELECT ... FOR UPDATE SKIP LOCKED`. Without it, multiple worker replicas race to claim the same items, producing duplicate processing and state corruption. E05-R-007 confirms this pattern works correctly.

### [PROCESS_CHANGE] Prometheus Metrics Must Use Dedicated `CollectorRegistry`
Any FastAPI service adding prometheus-client instrumentation must define `SERVICE_METRICS_REGISTRY = CollectorRegistry()` in a dedicated `metrics.py` module and register all metrics to it. The default global registry causes `ValueError: Duplicated timeseries` when pytest re-imports modules. This is a non-obvious failure that wastes significant debugging time.

### [PROCESS_CHANGE] Celery Task Failure Handlers Require Redis-Backed State, Not In-Memory Dicts
Any Celery `task_failure` signal handler that needs to correlate the failing task with a DB record (e.g., to update an audit row status) must store the correlation state in Redis (SETEX with TTL), not an in-memory dict. Prefork pool fires signals in a different OS process. E05 demonstrates the exact failure mode and the fix.

### [PROCESS_CHANGE] Non-Functional Stories with Evidence Requirements Must Not Close Without Artifacts
The NFR assessment flagged that the `load-test-results.md` in E12 S12.17 was an unfilled template (E12 anti-pattern). E05 S05.12 is the correct counter-example: the integration test suite produces real test output (102 tests, 245s runtime) and the NFR report contains real evidence. Apply this standard uniformly.
