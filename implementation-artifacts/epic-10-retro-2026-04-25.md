## 4. Challenges and Growth Areas

### 4.1 S10.16 — 37 Action Items: Largest Review in Project History

Amelia (Developer): "S10.16 (Approval Pipeline & Bid Decision UI) generated 37 action items in review round 1 — 10 blocking, 24 major/minor. This is the largest review payload in project history by a significant margin."

Charlie (Senior Dev): "The blocking issues tell a story. History table rendered in DESC order when spec mandated ASC. Override form didn't pre-populate from prior decision. `isUpdate` logic was broken. Global `dataUpdateCount` was racy. Polling never resumed after tab-hidden. None of these are small gaps."

Dana (QA Engineer): "And 11 polish items remain as tech debt post-approval. The story had 5,500+ LOC across pages, components, hooks, API clients, and ATDD."

Amelia (Developer): "The root cause: S10.16 covers three distinct workflows (approval stepper, bid/no-bid scorecard + radar chart, outcome recording + async poll). It should have been three stories. Monolithic frontend stories with heterogeneous business logic inevitably produce review debt."

**Lesson:** Frontend stories covering 3+ distinct user workflows should be split before entering the dev queue. A story with expected LOC > 3,000 is a signal for scope review.

### 4.2 S10.09 — SQL Injection Risk via String Interpolation

Charlie (Senior Dev): "S10.09 (Approval Decision Engine) initially constructed SQL with raw string interpolation on UUIDs: `WHERE stage_id = '{stage_id}'`. That's an injection risk. SQLAlchemy's text() with bindparams must be used for any dynamic SQL."

Amelia (Developer): "The review caught it and it was fixed. But it should not have reached review. The ruff linter doesn't catch this pattern; it requires code review scrutiny."

**Lesson:** Any story implementing query logic outside the ORM's query builder (raw `text()` clauses) must be flagged in ATDD as requiring an explicit injection test: `test_{feature}_sql_parameter_is_bound`. The test asserts the function uses `bindparam()` or named parameters, not f-strings.

### 4.3 S10.03 — 6 Blocking Issues in Round 1

Charlie (Senior Dev): "Section locking had six blocking issues in the first review: Celery cleanup task not decorated with `@app.task`; `scalar_one()` on the release endpoint (500 on missing row — should be `scalar_one_or_none()`); unbounded recursion in `acquire_lock`; INNER JOIN dropping orphaned lock records; naive datetime in cleanup comparison; AC16 snapshot not updated."

Elena (Junior Dev): "The recursion bug and the INNER JOIN bug are both correctness failures that would be invisible in happy-path unit tests. You only find them under contention or expiry edge cases."

Amelia (Developer): "The `@app.task` omission is the most predictable class of Celery bugs. It will fail silently at runtime — the function is never registered in the worker. We need a structural test pattern for this."

**Lesson:** Any story registering Celery tasks must include a test asserting task registration: `assert cleanup_task in current_app.tasks`. This is a one-line structural test that prevents silent no-op deployment.

### 4.4 S10.04 — Contradictory Spec Constraint

Dana (QA Engineer): "S10.04 (Comments API) had a contradictory AC: a CHECK constraint requiring `resolved_by IS NOT NULL` when `resolved = TRUE` — but a FK with `ON DELETE SET NULL` on `resolved_by`. If the resolver's user account is deleted, the row violates its own constraint."

Amelia (Developer): "The fix was to remove the NOT NULL part of the check constraint. But the fix required implementation to stop and resolve the contradiction. This is spec-review work that should happen before the story enters dev."

**Lesson:** Story ACs that reference ON DELETE FK behavior must be cross-validated against CHECK constraints before story creation. Add a "constraint consistency check" step to the story creation template for any story with FK + CHECK constraint interactions.

### 4.5 S10.10 — HTTP Status Code Violation

Amelia (Developer): "S10.10 (Bid/No-Bid Decision) initially always returned HTTP 201 on both insert and update. RFC 7231 requires 200 for updates. The fix: the service layer returns a `(response, was_insert)` tuple; the router uses 201/200 accordingly."

Charlie (Senior Dev): "This is a subtle but important correctness issue for API clients that parse status codes. 201 means 'resource created'; 200 means 'resource exists and was updated'. Idempotent clients behave differently."

**Lesson:** All upsert endpoints must have an explicit test asserting `201` on first call and `200` on second call with identical payload. This is now a required ATDD test pattern for any endpoint implementing INSERT ON CONFLICT.

A TEA review is the only mechanism that catches spec-vs-implementation constant contradictions that unit tests and ATDD both miss."

## 5. Key Insights and Learnings

### 5.1 Traceability-First Produces the First TRACE_GATE PASS

Running `bmad-testarch-trace` before implementation and maintaining a per-AC coverage map throughout the epic is the causal factor for TRACE_GATE PASS. The traceability matrix was the planning document, not an afterthought. Every story with an ATDD checklist mapped explicitly to the AC being covered.

**Standard:** `bmad-testarch-trace` output must exist before the first story in an epic begins development. The matrix is updated as stories complete.

### 5.2 Frontend Monolith Stories Create Review Debt

S10.16's 37 action items correspond to three distinct user workflows bundled into one story (approval stepper, bid/no-bid, outcome). The correlation is systematic: every frontend story in this project with 3+ distinct UI surfaces has generated ≥20 review items.

**Standard:** Frontend stories covering more than 2 distinct user workflows (distinct page/modal combinations, distinct API domains) must be split before entering the dev queue. Maximum expected LOC guideline: 3,000 LOC per frontend story.

### 5.3 SQL Injection Risk Requires Structural Test Coverage

SQL injection via string interpolation in SQLAlchemy `text()` clauses is not detected by ruff, mypy, or standard ATDD. It requires explicit structural tests asserting that dynamic values are bound parameters, not f-string interpolations.

**Standard:** Any story using SQLAlchemy `text()` or raw SQL for dynamic query building must include `test_{feature}_uses_bound_params` asserting via source inspection (or by attempting a value with SQL metacharacters and asserting 200/422, not 500).

### 5.4 Celery Task Registration is Invisible Without a Structural Test

A function missing `@app.task` will be called at definition time on task invocation but will never be registered in the worker registry. The call will succeed in unit tests (the function exists), fail silently in production (the worker has no task registered). This is the same class of invisible bug as the E09 S09.04 four-round review.

**Standard:** Every story introducing a Celery task must include: `assert "service.module.task_name" in celery_app.tasks` in its ATDD checklist. Zero exceptions.

### 5.5 Spec Consistency Is Pre-Story Work

S10.04's CHECK constraint + ON DELETE FK contradiction required mid-implementation correction. S10.03's Celery task omission could have been caught by a pre-story checklist. Spec review before implementation begins is cheaper than review-round correction.

**Standard:** The story creation template must include a "pre-implementation consistency checklist": (1) FK + CHECK constraint interactions consistent, (2) All Celery tasks present in ACs are decorated, (3) HTTP status codes match RFC 7231 for insert vs update.

### 5.6 Proposal-Level RBAC as a Blocking Gate Pattern

S10.02 (proposal-level RBAC) was implemented and verified before any story depending on it (S10.03–S10.11) was merged. This dependency ordering prevented a class of subtle authorization bypass bugs where features go live before their security layer. The pattern: security middleware stories are P0 blocking gates; no downstream story can enter `review` status until the middleware story is `done`.
