## 3. What Went Well

Alice (Product Owner): "I'm genuinely proud of how the billing architecture came together. Every billing concern is in one place — `client-api/services/billing/`. No other service needs to know Stripe exists."

Charlie (Senior Dev): "The VIES implementation is the pattern I'm most pleased with. It took discipline to make it fail-open. The easy path would have been to block registration on VIES failure — but that would have created a hard dependency on an external EU service with notoriously spotty uptime."

### [PATTERN] Clean Billing Service Boundary Architecture
`IMPACT: standards_update` `SEVERITY: medium`

Epic 8 establishes a clean, auditable billing layer: `billing_service.py`, `webhook_service.py`, `vies_service.py`, `tier_gate.py`, `tier_cache.py`, `api/v1/billing.py`. No cross-service DB joins; all events via Redis Streams; 100% isolated in `client-api/services/billing/`. **Reference architecture for all payment-adjacent services in subsequent epics.**

### [PATTERN] VIES Fallback-to-Pending Prevents Registration Blocking
`IMPACT: standards_update` `SEVERITY: high`

S08.10 correctly implements: VIES SOAP 503/timeout → `vat_validation_status: pending` (never block registration), retry on next billing event, reverse-charge applied only on `status: valid`. Three integration tests confirmed GREEN. **This is the mandatory pattern for any external validation service on the registration critical path.**

### [PATTERN] Stripe Customer Provisioning as BackgroundTask (Fail-Open)
`IMPACT: standards_update` `SEVERITY: high`

S08.01 provisions the Stripe customer in a FastAPI `BackgroundTask` after `POST /auth/register` returns 201. `stripe_customer_id` is nullable until provisioned; all billing endpoints return 422 with a clear error if customer not yet created. **Apply to any Epic's integration with external payment/CRM services.**

### [PATTERN] Webhook Dedup Table as Idempotency Layer
`IMPACT: standards_update` `SEVERITY: high`

S08.04 uses a `webhook_events` table with a `stripe_event_id` unique constraint as the canonical idempotency mechanism. Pattern: INSERT inside the processing transaction; unique-constraint violation → return 200. **Not `on_conflict_do_nothing` or in-memory caching — database constraint is the only reliable mechanism.**

### [PATTERN] Tier Cache DELETE-not-SET for Event-Driven Cache Invalidation
`IMPACT: standards_update` `SEVERITY: medium`

S08.14 uses `tier_cache.delete(company_id)` — not `set` — when a `subscription.changed` event is processed. `DELETE` forces the next request to read the authoritative DB value. `SET` creates a read-before-write race. **DELETE-not-SET is mandatory for all event-driven cache invalidation in the platform.**

### [PATTERN] `asyncio.to_thread()` Mandatory for Sync SDK Calls in Async FastAPI
`IMPACT: standards_update` `SEVERITY: high`

S08.01 wraps synchronous `stripe.*` SDK calls with `asyncio.to_thread()`. The Stripe Python SDK is synchronous; calling it directly in `async def` blocks the event loop. **This pattern extends E07's ThreadPoolExecutor requirement (CPU-bound) to I/O-bound synchronous SDKs (Stripe, VIES SOAP, legacy REST clients).**

### [PATTERN] NFR PASS Without Hardening Story Injection — Security Architecture Maturing
`IMPACT: standards_update` `SEVERITY: medium`

Unlike E07 (which required S07.17 injection to fix two NFR FAIL items), E08's NFR assessment produced PASS (with CONCERNS) on the first run. Billing architecture correctly applied E07 security hardening lessons (HMAC verification, idempotency, GDPR data processing) from the start.

## 9. Lessons for Subsequent Epics

Charlie (Senior Dev): "The single biggest lesson from Epic 8 is that the retro-to-action loop is broken. We're generating structured findings, they're being documented — and then nothing happens. The Orchestrator needs to enforce verification."

Elena (Junior Dev): "For me personally, the VIES fail-open pattern is the takeaway. Defensive coding around external service downtime — build for failure, not for success."

Alice (Product Owner): "Epic 8 showed we can ship complex billing infrastructure cleanly when the architecture is right. The billing service boundary is something I'll reference for years."

**Summary of transferable learnings:**

1. **Any external service integration** must be designed to fail-open with a `pending` state
2. **Payment webhook processors** require a dedup table at the DB constraint level — no shortcuts
3. **Synchronous SDKs in async FastAPI** always need `asyncio.to_thread()` wrapping
4. **Cache invalidation on events** must DELETE, not SET — avoids read-before-write races
5. **Retrospective [ACTION] items with SEVERITY: critical** need verification tasks, not just documentation
6. **E2E specs written as `test.skip`** provide zero quality coverage — not PARTIAL, NONE
