## Part 1: Epic 3 Review

### What Went Well

---

#### [PATTERN] All Three Score-6 Risks Fully Mitigated
**IMPACT: standards_update | SEVERITY: high**

Three architectural risks that could have caused silent security or reliability failures were resolved with targeted test investment:
- **E03-R-001** (locale routing loop) — `locale-redirect.spec.ts` with `countRedirects()` asserting ≤1 hop
- **E03-R-002** (route guard flash) — Dual-layer `<AuthGuard>` + `middleware.ts`; opaque spinner during hydration
- **E03-R-003** (401 refresh deduplication) — Promise singleton refresh-lock; prevents E02 token family revocation

These became the canonical patterns for all subsequent epics.

---

#### [PATTERN] `packages/ui` as Platform Foundation
**IMPACT: standards_update | SEVERITY: high**

46+ shadcn/ui components, Zustand stores (`authStore`, `uiStore`), `apiClient`, `useSSE`, `useZodForm`, `<FormField>`, `<AppShell>`, `<QueryGuard>`, skeleton variants, `<ErrorBoundary>`, `<EmptyState>`, `<BottomNav>`, and `useToast()` all live in the shared library. Every feature epic (E04–E12) builds on this foundation without reinventing patterns.

---

#### [PATTERN] `<QueryGuard>` — Unified Data-Fetching State Machine
**IMPACT: standards_update | SEVERITY: high**

`<QueryGuard isLoading isError isEmpty>` wraps TanStack Query state into a deterministic state machine: loading → skeleton, error → error boundary, empty → empty state, success → children. Eliminates scattered conditional renders. Mandatory pattern for all list/table/detail views in E04+.

---

#### [PATTERN] `useZodForm(schema)` + `<FormField>` — Single Form Construction Pattern
**IMPACT: standards_update | SEVERITY: high**

Schema-first approach: define Zod schema in `lib/schemas/`, call `useZodForm(schema)`, compose `<FormField>` variants. Provides zodResolver wiring, animated inline error display, BG/EN translation, and RHF controller binding for free. All 9 type variants covered. No ad-hoc form construction acceptable.

---

#### [PATTERN] Server-Compatible Shell with Minimal `use client` Boundary
**IMPACT: standards_update | SEVERITY: medium**

`<AppShell>`, `<Sidebar>`, `<TopBar>` are server components. Only interactive leaves (`<SidebarToggle>`, `<UserAvatarMenu>`, `<LanguageSelector>`) carry `"use client"`. This preserves Next.js SSR performance while isolating interactive state. Feature components must follow the same `use client` boundary discipline.

**IMPACT: config_tuning | SEVERITY: medium**

The Run 1 TRACE_GATE failure (P0 at 90%, missing E03-P0-005) was resolved surgically: one spec file added, one redirect-count assertion, PASS in Run 2. Validates TRACE_GATE as an effective quality gate mechanism.

**IMPACT: prompt_adjustment | SEVERITY: high**

E03 introduced: `stub-company-id`, OAuth code exchange stubs, MSW handlers, `loginUser` stub responses, `setTimeout(800ms)` delay stubs. None have a formal removal checklist with owner and target epic. Silent integration failures are the primary risk for E04 API integration work.

#### [ANTI-PATTERN] `turbo: "latest"` — Non-Reproducible Builds
**IMPACT: config_tuning | SEVERITY: medium**

Root `frontend/package.json` uses `"turbo": "latest"`. Each fresh `pnpm install` may pull a different Turborepo major version. Currently on v2.9.5; a v3 release could silently break the build pipeline.

**Action:** Pin to `"turbo": "^2.9.5"` (or exact version) immediately.

### Process Discoveries

---

#### [ACTION] AuthGuard Hydration Timeout Required
**IMPACT: story_injection | SEVERITY: high**

Corrupt `localStorage` (e.g., manually edited JSON in `eusolicit-auth-store`) causes `onFinishHydration()` to never fire → permanent spinner with no recovery. NFR identified this as HIGH outstanding.

**Resolution:** Rule #24 in project-context.md: `setTimeout(() => { if (!hydrated) { setHydrated(true); setIsAuthenticated(false); } }, 5000)`.

---

#### [ACTION] Zustand Persist Keys Must Be Namespaced Per App
**IMPACT: standards_update | SEVERITY: high**

`eusolicit-auth-store` shared by both apps creates a collision risk if deployed under the same origin (e.g., behind a reverse proxy). Admin app reads client user tokens.

**Resolution:** Rule #25 in project-context.md: `eusolicit-client-auth-store` / `eusolicit-admin-auth-store`.

---

#### [ACTION] `useSearchParams()` Requires `<Suspense>` Boundary
**IMPACT: standards_update | SEVERITY: medium**

Next.js App Router throws a build error if `useSearchParams()` is called outside a `<Suspense>` boundary. Discovered in S03.08 (OAuth callback page). This pattern must be carried forward to any page reading URL search params (e.g., redirect paths, filter state, pagination).

**Resolution:** Wrap components using `useSearchParams()` in an inner `<Suspense>` boundary from the page default export. See S03.08 `callback/page.tsx` as the reference implementation.

---

#### [ACTION] `x-request-id` Correlation Header in apiClient
**IMPACT: standards_update | SEVERITY: medium**

No request correlation between frontend errors and backend logs. apiClient should inject `x-request-id: crypto.randomUUID()` on every request.

**Resolution:** Rule #27 in project-context.md. Must be verified live (currently in deferred-work.md).
