## 6. Key Insights and Learnings

### 6.1 Cross-Service Celery Task Dispatch: The Invisible Bug Class

The most expensive bugs in Epic 9 were cross-boundary: a consumer dispatches a Celery task, the dispatch call succeeds, but the worker task body fails with a type error or missing config. Unit tests that mock at the dispatch boundary can't catch these.

**New standard:** Any consumer story that dispatches a Celery task (via `send_email.delay`, etc.) MUST include at least one test that executes the real task body with a mocked transport layer (mocked `SendGridAPIClient`, etc.) â€” not just asserting `delay.called_once_with(...)`.

### 6.2 SQLAlchemy `default=` vs. Python-side UUID Generation

`mapped_column(default=uuid.uuid4)` fires at flush/INSERT, NOT at `__init__`. If the UUID is needed in the payload **before** the session commit (as required by the R-001 ordering guarantee), it must be explicitly pre-generated:

```python
# WRONG â€” id is None until flush
alert_log = AlertLog(user_id=user_id, ...)
payload["alert_log_id"] = str(alert_log.id)  # str("None") = "None"

# CORRECT â€” pre-generate UUID, pass to both
alert_log_id = uuid.uuid4()
alert_log = AlertLog(id=alert_log_id, user_id=user_id, ...)
payload["alert_log_id"] = str(alert_log_id)  # UUID string âś…
```

This applies to ANY scenario where the ORM object's ID must be referenced before the session is committed.

### 6.3 idempotency Ordering for Consumer Dispatch

The correct order for at-least-once delivery with no duplicate notification is:
1. Check idempotency guard (query for existing record)
2. Dispatch Celery task via `asyncio.to_thread(task.delay, ...)`
3. On dispatch success, commit AlertLog/idempotency record
4. ACK Redis stream message

If commit fails after dispatch, the task is already enqueued. On retry, idempotency guard fires and prevents duplicate. This is **dispatch before commit** for atomic-ish behaviour.

For batch tasks (digest), the ordering is reversed: collect payloads â†’ commit all records â†’ dispatch emails. This prevents sending emails whose DB records were never written.

### 6.4 Beat Schedule Constants Must Be Verified Pre-Story

S09.01 had a naming conflict: `sync-usage-to-stripe-daily` (S08.08's existing entry) vs. `sync-stripe-usage-metering` (S09.11's AC9 intent). The test was written against the intended name, the Beat schedule had the old name. The discrepancy only surfaced in Review Round 2 of S09.01.

**Standard:** Every story that modifies Beat schedule entries must cross-reference existing keys in `beat_schedule.py` before writing tests or implementation.
