### 4. [PATTERN] Materialized View Architecture is Production-Ready

**IMPACT: standards_update | SEVERITY: low (positive)**

The `REFRESH MATERIALIZED VIEW CONCURRENTLY` approach with unique indexes on filter columns is the correct pattern for analytics-at-scale. S12.01 verified: concurrent reads not blocked during refresh, rollback clean, and index presence enforced by integration test. This design handles production analytics workloads without read downtime.

[PATTERN] Any future analytics service using PostgreSQL materialized views must: (1) use `CONCURRENTLY` with unique index, (2) include a `TestE12DB006RefreshConcurrently`-equivalent test asserting no read timeouts during refresh, (3) include migration rollback test.

### 5. [PATTERN] CSV Streaming Export for Large Admin Datasets

**IMPACT: standards_update | SEVERITY: low (positive)**

`GET /admin/audit-logs/export` returns a `StreamingResponse` with `Content-Type: text/csv` to avoid memory pressure on large audit datasets. This is the correct pattern for paginated-but-large admin exports. The implementation avoids loading all rows into memory.

[PATTERN] Any admin endpoint returning potentially large datasets (>10k rows) must use streaming CSV export rather than in-memory JSON aggregation. Pattern: `StreamingResponse(generate_csv_rows(query), media_type="text/csv")`.

## Process Learnings

### 1. [PROCESS_CHANGE] Frontend Story ATDD Must Be a Hard Gate Before `done`

The complete absence of frontend E2E tests for 7 stories is not a one-time lapse — it's a systematic process gap. The `bmad-testarch-atdd` step was skipped for all frontend stories in E12. Going forward: **any story with frontend ACs requires a Playwright spec file (even if RED-phase) as a deliverable before the story can be marked `done`**. The spec file must exist before `done`, and must be activated (GREEN) before the epic retrospective.

### 2. [PROCESS_CHANGE] Non-Functional Stories (S12.17 Pattern) Need Pre-Execution Scaffolding

S12.17 failed because the output artifacts (load test results, OWASP checklist) require external execution that was never triggered. Future performance/security stories must have entry criteria that verify: (1) staging environment available, (2) tooling (k6, ZAP, LocalStack) confirmed available, (3) story is not marked `done` until evidence files contain actual results (not templates).

### 3. [PROCESS_CHANGE] TEA Review Concurrently with Story Implementation, Not at Epic Close

Running TEA at epic retrospective time is too late — the retrospective identified 0 of 18 stories with TEA reviews. TEA must gate `review → done` per story. The `tea_status` field in sprint-status.yaml must be updated story-by-story.

### 4. [PROCESS_CHANGE] RED-Phase ATDD Specs Require Owner + Activation Story at Creation Time

Every RED-phase spec file must have a corresponding story in the sprint backlog with explicit ACs for removing `test.skip()` and achieving GREEN. "Deferred to TEA phase" without a named story is not acceptable.

### 5. [PROCESS_CHANGE] Recharts/Analytics Frontend Pattern Needs Explicit E2E Assertions

The analytics frontend stories (S12.03–S12.08) use Recharts + TanStack Query + `<QueryGuard>`. Future analytics stories must codify the E2E assertion pattern: `<svg>` element present, ≥1 data point rendered, hover tooltip fires on `mouseover`, `<EmptyState>` shown when data array is empty. Chart rendering failures are invisible to Python API tests.
